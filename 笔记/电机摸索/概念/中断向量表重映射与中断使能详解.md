# `__enable_irq()` 和 `RCC_FLASHRemapConfig()` 详解

## 一、函数概述

### 1.1 `RCC_FLASHRemapConfig(_VECTOR_TABLE_OFFSET_, ENABLE)`

**定义位置：** `Libraries/单片机外设驱动库/src/外设复位配置.c:675`

**函数原型：**
```c
void RCC_FLASHRemapConfig(uint16_t FlashAddrOffset, FunctionalState NewState)
{
    if (NewState != DISABLE)
    {
        RCC->REMAPFLASH = RCC_REMAP_FLASH_ENABLE | FlashAddrOffset;
    }
    else
    {
        RCC->REMAPFLASH = RCC_REMAP_FLASH_DISABLE;
    }
}
```

**参数定义：**
```c
// function_config.h:9
#define _VECTOR_TABLE_OFFSET_ (0x1400)  // addr:0x08001400
```

---

### 1.2 `__enable_irq()`

**定义位置：** `Libraries/CMSIS/Include/cmsis_iccarm.h`

**宏定义（IAR编译器）：**
```c
#define __enable_irq __iar_builtin_enable_interrupt
```

**实际作用：**
```c
// 汇编指令
CPSIE I  // Clear PRIMASK, enable interrupts
```

## 二、为什么需要这两个函数？

### 2.1 中断向量表重映射

#### **什么是中断向量表？**

中断向量表是一个函数指针数组，存储了所有中断处理函数的地址：

```
地址          内容
0x08000000  → 初始栈指针 (SP)
0x08000004  → Reset_Handler
0x08000008  → NMI_Handler
0x0800000C  → HardFault_Handler
...
0x08000140  → TIMER0_CAP_IRQHandler
...
```

#### **为什么需要重映射？**

**问题：**
```
默认情况：
- 代码起始地址：0x08000000
- 向量表位置：  0x08000000
- 中断处理：    查找 0x08000000 开始的向量表

但实际项目中：
- Flash起始：   0x08000000
- Bootloader：  0x08000000 - 0x080013FF (5KB)
- 应用程序：    0x08001400 开始
- 应用向量表：  0x08001400 开始

如果不重映射：
- CPU去0x08000000找向量表 → 找到Bootloader的向量表 → 调用错误的中断处理函数
```

**解决：重映射向量表**
```c
RCC_FLASHRemapConfig(_VECTOR_TABLE_OFFSET_, ENABLE);
// _VECTOR_TABLE_OFFSET_ = 0x1400

// 配置后：
// CPU去 0x08000000 + 0x1400 = 0x08001400 找向量表
// → 找到应用程序的向量表 → 正确调用中断处理函数
```

---

### 2.2 全局中断使能

#### **Cortex-M0的中断控制**

```
复位后：
├── PRIMASK寄存器 = 1  → 中断禁用（除NMI和HardFault）
└── 所有外设中断被屏蔽

调用 __enable_irq() 后：
├── PRIMASK寄存器 = 0  → 中断使能
└── 外设中断可以正常响应
```

#### **为什么需要？**

**问题：如果不使能中断**
```c
SysTick_Init();     // 配置了SysTick定时器
Timer0_Init();      // 配置了定时器0
MCPWM_Init();       // 配置了电机控制PWM
// 但中断没有使能！

// 结果：
// - SysTick不产生中断 → Delay_ms()无法工作
// - Timer0中断不触发 → 无法测量转速
// - PWM中断不触发 → 无法换相
// - 电机无法运行！
```

**解决：使能中断**
```c
__enable_irq();  // PRIMASK = 0
// 现在所有配置好的中断都能正常工作
```

---

## 三、能去掉吗？

### ❌ **强烈建议不要去掉！**

| 函数 | 能否去掉 | 原因 |
|------|---------|------|
| `RCC_FLASHRemapConfig()` | ❌ **不能** | 没有它，中断会跳转到Bootloader的处理函数，系统崩溃 |
| `__enable_irq()` | ❌ **不能** | 没有它，所有中断都不工作，系统无法运行 |

---

## 四、去掉的后果

### 4.1 去掉 `RCC_FLASHRemapConfig()`

**现象：**
```
1. 系统上电
2. 定时器中断发生
3. CPU去 0x08000000 找向量表
4. 找到Bootloader的向量表（不是应用程序的）
5. 跳转到Bootloader的中断处理函数
6. Bootloader的中断处理函数可能：
   - 做了错误的处理 → 系统异常
   - 或者直接返回 → 中断丢失
   - 或者跳转到错误的地址 → 程序跑飞
```

**错误示例：**
```
期望：
Timer0中断 → TIMER0_CAP_IRQHandler → 测量转速

实际（去掉重映射后）：
Timer0中断 → Bootloader的Timer0处理 → 未知行为/系统崩溃
```

---

### 4.2 去掉 `__enable_irq()`

**现象：**
```
1. 系统初始化完成
2. 主循环开始运行
3. 定时器计数到 → 中断标志位置位
4. CPU检查PRIMASK = 1 → 中断被屏蔽
5. 中断请求被忽略
6. 结果：
   - SysTick不工作 → Delay_ms()卡死
   - Timer0中断不触发 → 无法测量转速
   - PWM中断不触发 → 无法换相
   - UART中断不触发 → printf无法输出
   - 系统完全无法工作
```

## 五、这些函数来自哪里？

### 5.1 来源

#### **`RCC_FLASHRemapConfig()`**
- ✅ **不是芯片手册直接提供的**
- ✅ **是单片机标准外设库提供的**
- ✅ **基于芯片的REMAPFLASH寄存器实现**

**芯片手册章节：**
- 参考手册 → RCC寄存器 → REMAPFLASH寄存器
- 说明如何通过寄存器配置向量表重映射

#### **`__enable_irq()`**
- ✅ **是ARM Cortex-M内核指令**
- ✅ **由CMSIS标准定义**
- ✅ **编译器内置函数**

**ARM文档：**
- ARM Cortex-M0 技术参考手册
- PRIMASK寄存器说明
- CPS指令说明

---

### 5.2 标准代码模板

#### **ARM标准启动流程（CMSIS）：**

```c
// startup_cortexm.s（启动文件）
Reset_Handler:
    LDR     R0, =SystemInit
    BLX     R0          // 调用SystemInit()

    LDR     R0, =__iar_program_start
    BX      R0          // 跳转到C程序

// main.c
int main()
{
    SystemInit();        // 通常由启动文件调用
    // 如果没有调用，需要手动配置
    __enable_irq();      // 使能中断
    while(1) { ... }
}
```

## 七、特殊情况分析

### 7.1 什么时候可以去掉？

#### **情况1：没有Bootloader**

```c
// 应用程序直接从0x08000000开始
#define _VECTOR_TABLE_OFFSET_ (0x0000)

// 可以去掉重映射，但保留也无害
RCC_FLASHRemapConfig(0x0000, ENABLE);  // 重映射到起始地址
```

#### **情况2：使用IAR的向量表自动重定位**

```c
// IAR启动文件中可能已经处理
// 检查 启动文件.s
// 如果有类似代码：
//   LDR VTOR, =0x08001400
//   MSR VTOR, VTOR
// 则可以去掉 RCC_FLASHRemapConfig()
```

---

### 7.2 什么时候必须保留？

| 场景 | 必须保留 | 原因 |
|------|---------|------|
| **有Bootloader** | ✅ 是 | 应用程序向量表不在0地址 |
| **使用中断** | ✅ 是 | 没有中断系统无法工作 |
| **使用RTOS** | ✅ 是 | RTOS依赖中断进行调度 |
| **使用HAL库** | ✅ 是 | HAL库大量使用中断 |

## 八、总结

### 8.1 核心结论

| 问题 | 答案 |
|------|------|
| **能去掉吗？** | ❌ **不能** |
| **为什么需要？** | 重映射向量表、使能中断 |
| **来自哪里？** | CMSIS标准、芯片厂商库 |
| **是否可优化？** | 无需优化，已是最简 |

### 8.2 作用对比

| 函数 | 作用对象 | 影响范围 | 错误后果 |
|------|---------|---------|---------|
| `RCC_FLASHRemapConfig()` | 中断向量表位置 | 所有中断 | 系统崩溃/中断异常 |
| `__enable_irq()` | PRIMASK寄存器 | 全局中断 | 系统完全无法工作 |

### 8.3 最佳实践

```c
int main()
{
    // ✅ 推荐：保持标准顺序
    RCC_FLASHRemapConfig(_VECTOR_TABLE_OFFSET_, ENABLE);  // 1. 重映射向量表
    __enable_irq();                                        // 2. 使能中断

    // 其他初始化
    SysTick_Init();
    STimer_Init();
    // ...

    while (1) { ... }
}
```

**原因：**
- 这是ARM Cortex-M的标准启动流程
- 由ARM和芯片厂商共同定义
- 修改会破坏系统正常工作

## 十、常见问题

### Q1: 我的程序没有Bootloader，还需要这个吗？

**A:** 如果应用程序从 0x08000000 开始，可以将偏移设为 0：
```c
#define _VECTOR_TABLE_OFFSET_ (0x0000)
RCC_FLASHRemapConfig(0x0000, ENABLE);
```
但**建议保留**，因为：
- 代码移植方便
- 将来可能添加Bootloader
- 不会有负面影响

---

### Q2: 可以在其他位置调用这两个函数吗？

**A:** 可以，但建议：
- ✅ **main()开始时调用**（当前做法）
- ⚠️ 中间修改可能导致中断错乱
- ❌ 不要在ISR中调用

---

### Q3: 如何验证配置是否正确？

**A:** 检查以下几点：
```c
// 1. 检查向量表位置
//    → 链接器.map文件中 __vector_table 地址

// 2. 检查偏移值匹配
//    → _VECTOR_TABLE_OFFSET_ 与链接器配置一致

// 3. 测试中断是否工作
void TIMER0_CAP_IRQHandler(void)
{
    static int count = 0;
    count++;  // 设置断点，看是否能进入
}

// 4. 检查VTOR寄存器（如果支持）
uint32_t vtor = SCB->VTOR;
printf("VTOR = 0x%08X\n", vtor);  // 应该是 0x08001400
```
